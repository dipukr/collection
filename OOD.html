Object Oriented Programming
Object oriented programming is way of programming oriented towards the objects. An object oriented program is basically a collection of objects and relationship and communication among them. Related data and code that operate on that data is considered as object. Four main concept of OOPs are abstraction, encapsulation, inheritance and polymorphism which help us to write modular, reusable, extensible and generic software system.

Object is a concrete entity with a set of attributes and a set of behaviours. Object also have identify, lifetime and scope and belongs to a class. Fields defines the attributes of objects instantiated from a class. Methods defines the behaviors of objects instantiated from a class. Objects maintain state in instance variables (attributes). Objects are allocated from the heap and accessed by reference variables. Objects communicates with one another by sending and receiving message.

A class is a blueprint which defines the attributes (fields) and behaviours (methods) common to a number of objects. A class is a user defined blueprint based on which objects are created. Class is the unit of modularization in object oriented program. A class is defined using class keyword. A class body contains contains specification of a set of fields which defines attributes of the objects and a set of methods which defines behaviours of the objects.

Encapsulation
Encapsulation refers to the bundling of data with the mechanisms or methods that operate on the data. Encapsulation also refer to the limiting of direct access to some of that data, such as an object's components. Essentially encapsulation prevents external code from being concerned with the internal workings of an object. Encapsulation allows developers to present a consistent interface that is independent of its internal implementation. A language mechanism for restricting direct access to some of the object's components. A language construct that facilitates the bundling of data with the methods (or other functions) operating on those data.

Encapsulation is the process of wrapping of related data and code into a single unit which modularizes our software system. Encapsulation also help us to achieve data hiding which is the process of hiding sensitive data of the objects from the outside world. Encapsulation is java is achieved through class. Encapsulation makes out software system modular ie. we can change one part of the code without affecting other parts of code.

Data hiding is the process of hiding sensitive data of the object from the user of the object. Data hiding is a programming concept which protects the data from direct modification by other parts of the program. Data hiding is achieved by declaring fields as private and providing setter and getter methods for each field to access and update the value. Data hiding is closely related to abstraction and encapsulation. Data hiding is achieved through encapsulation.

Advantages of encapsulation:
  • Data security: help us to make fields of a class read-only or write-only.
  • Encapsulation gives a clear and modular structure to our software system.
  • Program is well organized so easy to maintain.
Inheritance
Inheritance is a mechanism by which one class acquires the attributes (fields) and functionalities (methods) of another existing class. Inheritance enable us to create modules that does not need to be changed when new attributes or functionalities is added to the class. Inheritance helps to create reusable and extensible software components. Reusability helps us to avoid duplicate code and data. Inheritance helps us to define relationship among classes.

Kind of inheritance:
  • Single inheritance - when a class inherits another class.
  • Multi-level inheritance - when a class inherits derived class.
  • Hierarchical inheritance - when a class is inherited by many classes.
Abstraction
Abstraction is a mechanism of hiding unnecessary details and showing only essential details to the user by separating interface from its implementation. Abstraction also generalises by capturing the common essense among unrelated classes. Main goal of the abstraction it to reduce the complexity of the design and implementation process of software system. Abstraction can be achieved through interface and abstract class.

Abstract method is a method that is declared without implementation. Abstract method must always be redefined in the subclass. Abstract method only have signature. Abstract method can not be final. Class which contains abstract method must be declared abstract.

Abstract class is one of two ways to achieve abstraction in Java. Abstract class declared with abstract keyword. Abstract class cannot be instanciated. Abstract class cannot be final. Abstract class always have a default constructor. Abstract class can have parameterized constructors. Abstract class is created to be extended or inherited. Abstract class can have abstract and non-abstract methods. A class that extends an abstract class must implement all its abstract method.

Interface is one of two ways to achieve abstraction in java. Interface is completely abstract class. Interface methods are by default abstract and public. Interface variables are by default public, static and final. Interface specifies only what needs to be done. Childrens will decide how to do in their own way. Like abstract class interface cannot be instanciated. Interface allows us to group several related classes as siblings.

Method = Method header + method body. Method header is an abstraction of a block of code. Abstract method is independent of implementation which can have many implementations.

Abstraction: Complete abstraction can be achieved through interface.
Security: Hides implementation details and only show the important details of an object.
Multiple inheritance: Can be achieved through interface because a class can implements multiple interfaces.
Polymorphism
Polymorphism refers to one-to-many relationship, which allows an entity such as an variable, a method or an object to have more than one form. Polymorphism enables us to deal with generalities and let the compiler and runtime environment to handle the specific. Polymorphism in Java can be achieved through method overloading, method overriding and Generics. Polymorphism is the provision of a single interface to entities of different types or the use of a single symbol to represent multiple different types. Field level polymorphism: a reference variable of class T can point to object of class E as well as object of subclass of E.

  • Many methods can have one common name.
  • One method can accept many types of arguments.
  • One method header can be associated with many method body.
  • One type of reference variable can be used to access many type of objects.
Method-overloading
Process of writing multiple methods with the same name in the same class is called method overloading. Method-overloading is a kind of static polymorphism.

  • Method overloading increases the readability of the program.
  • Avoids the need to memorize multiple method names.
  • Method overloading makes our code more generic.
  • Method overloading support clean code.
  • The complexity of the code is reduced.
  • Binding process happens during compile time.
  • Compiler binds the method call with an approriate method body.

Method overloading is achieved by:
  • Changing the number of arguments.
  • Changing the data type of the arguments.
Method-overriding
Method-overriding is the process of redefining a parent class instance method into its child class is known as method overriding. An instance method defined in the parent class is redefined into its child class to give specific implementation of the parent class method.

Referenced object type, not the reference variable type determines which method will be called. When we create a parent class reference variable and assign a child class object then calling an overridden method on that variable will call the child class behaviour.

  • Final methods can not be overridden.
  • Static methods can not be overridden.
  • Private methods can not be overridden.
  • Child class method is called overriding method.
  • Parent class method is called overridden method.
  • Overridden methods are dynamically dispatched.
  • Finalize method can be overridden to free system resources.
  • Overridden parent class method can be called using super keyword.
  • Runtime polymorphism can be achieved through method overriding.
  • Abstract methods can only be overridden by the first concrete class.
  • The access level cannot be more restrictive than the overridden method.
  • Override annotation denotes that the method overrides the base class method.

Dynamic binding is a process in which a call to overridden method is resolved at the runtime rather than compile time. Which means that the code to be executed for a specific method call is not known until runtime. Dynamic dispatch is used when multiple classes contain different implementations of the same method which provides a mechanism for selecting the function to be executed from various function alternatives at the runtime. Dynamic binding (dispatch) is the process of linking function call to a specific sequence of code (function) at run-time.
Dependency inversion
Dependency inversion principle (DIP) states that high level modules should not depend on low level modules; both should depend on abstractions. Abstractions should not depend on details rather details should depend on abstractions.
Composition
Process of building complex objects from simpler ones is called object composition. Composition is the design technique in object-oriented design to implement has-a relationship between objects. Composition supports code reuse.

Benefits of composition:
  • Increases reusability and flexibility.
  • Supports dependency injection and strategy pattern.
  • Enables plug-in architectures (swap modules dynamically).
  • Reduces class explosion (no need for hundreds of subclasses).
Composition vs Inheritance
Both composition and inheritance used to solve the same problem that is code reuse. Process of building complex objects from simpler ones is called object composition.
SOLID
Solid principle is a set of guidelines for writing code that makes it easier to manage, extend and debug. Solid principle help us break big problem into smaller easier to handle parts. Solid reduces dependencies on code so changes in one area won't break another. Solid reduces tight coupling, making code more flexible and easier to maintain.
DI
Dependency injection generally means passing a dependent object as a parameter to a method rather than having the method create the dependent object. Dependency injection is a software design pattern that allow us to develop loosely-coupled software. Dependecy injection allows us to depends more on abstraction rather than the concrete implementation, which promotes loosely-coupled architecture, flexibility and pluggability within our code. What it means in practice is that the method does not have a direct dependency on a particular implementation, any implementation that meets the requirements can be passed as a parameter.
Design Patterns
Design patterns are solutions or techniques to avoid code design problem. Design patterns help us write robust, clean, modular and extensible code. Design patterns are reusable solutions to commonly occurring problems in software design. They provide a template for solving design issues in various contexts, helping developers write code that is more maintainable, scalable and understandable.

Design patterns are divided into three categories:
  • Creational patterns - Deals with object creation.
  • Structural patterns - Deals with object composition.
  • Behavioral Patterns - Deals with object interaction and responsibility.

Reusability: Design patterns provide a reusable solution to common software design problems. This saves developers from reinventing the wheel whenever they encounter a new problem. Instead, they can use a tried and tested pattern to solve the problem.

Maintainability: Design patterns promote clean, organized and maintainable code. This makes it easier for developers to maintain and modify the code as the software evolves.

Rapid Development: Design patterns provide a standard way of solving common software design problems. This saves time and effort in the desing and development process, allowing developers to focus on other aspects of the software.

Creational: Singleton, Builder, Factory Method, Abstract Factory and Prototype.
Structural: Adapter, Bridge, Composite, Decorator, Facade, Flyweight and Proxy.
Behavioral: CoR, Command, Iterator, Mediator, Memento, Observer, State, Strategy, Template and Visitor.
Builder
Builder design pattern is a creational design pattern that separates the construction of a complex object from its representation. Builder design pattern allows the same construction process to create different representations of the object.
Adapter
Adapter pattern is a structural design pattern that allows objects with incompatible interfaces to work together. It acts as a bridge between two incompatible interfaces. Adapter pattern converts the interface of a class into another interface that the client expects. It lets classes work together that couldn’t otherwise because of incompatible interfaces. It catches calls for one object and transforms them to format and interface recognizable by the second object.

Adapter pattern should be used when:
  • There is an existing class and its interface does not match the one you need.
  • We need to connect systems or components that were not built to work together.

Components:
  • Target: The interface the client expects.
  • Adaptee: The existing class that needs adapting.
  • Adapter: The class that implements the target interface and internally uses or adapts the adaptee.

Pros and Cons:
  • Reusability of existing functionality.
  • Increased flexibility and compatibility.
  • Keeps existing code untouched.
  • Code can become complex if many adapters are introduced.
  • Sometimes it's better to refactor than use adapters excessively.
Bridge
Bridge pattern decouples an abstraction from its implementation so that the two can vary independently.
Facade
Facade pattern is a structural design pattern that provides a unified interface to a set of interfaces in a subsystem. Facade defines a higher-level interface that makes the subsystem easier to use. A complex interface to the client is considered as a fault in the design of a system. Facade pattern makes a complex interface easier to use using a Facade class. Banking system: we interact with a banking mobile app or web UI (Facade) instead of all internal services like account service, fund transfer, loan service, etc. Hotel reception: receptionist (Facade) handles room booking, food service, taxis, etc. instead of the customer contacting each department.

When-to use:
  • When you want to simplify the usage of a complex system.
  • When there are multiple subsystems that need to be integrated.
  • To decouple clients from the internal components of a system.
  • When you want to create a layered architecture.

Advantages:
  • Simplifies complex interfaces.
  • Makes the system easier to use.
  • Promotes loose coupling.
  • Improves code readability.
Composite
Composite pattern is a kind of structural design pattern used when we want to treat individual objects and compositions of objects uniformly. Composite pattern is especially useful when you deal with hierarchical tree structures, such as file systems, DOM tree or UI components. Composite lets clients treat individual objects and compositions of objects uniformly. When-to use: when we want to compose objects into tree structures to represent part-whole hierarchies.

Components:
  • Component: interface or abstract class for all objects in the hierarchy, both composite and leaf.
  • Leaf: represents individual objects (end nodes). Implements the base Component.
  • Composite: a container that holds child Components. Can be other Composites or Leafs.
  • Client: uses the Component interface to interact with objects in the hierarchy uniformly.

Use cases:
  • GUI frameworks (Swing or HTML DOM trees)
  • Organization hierarchy (employees, managers)
  • File system (files and directories)
  • Document structures (paragraphs, sections, characters)
Proxy
Proxy pattern is a structural design pattern that provides a proxy for other objects to control access to them. The proxy implements the same interface as the real object and can add behavior like access control, logging, caching, lazy loading, etc. A proxy functions as an interface to something else. A proxy is a wrapper or agent object that the client calls to access the real serving object behind the scenes. Extra functionalities can be provided into proxy such as caching the operations on the actual object or checking pre-conditions before calling operations on the actual object. When-to use: when we need to perform additional operations when accessing an object, for examples, lazy loading, caching and access control.

Components:
  • Subject (Interface) – Declares common interface for RealSubject and Proxy.
  • RealSubject – The actual object that does the work.
  • Proxy – Controls access to RealSubject, implements the same interface.

Pros and Cons:
  • Adds a layer of control over real objects.
  • Promotes separation of concerns.
  • Adds extra complexity and indirection and performance overhead.

Real-world example:
  • Java RMI – Uses proxy to call remote objects.
  • Spring AOP – Uses proxies for method interception.
  • Hibernate – Lazy-loaded entities are proxies.
  • Security managers – Proxy access to critical resources.
  • Credit card - Acts as a proxy for our bank account.

Types of Proxy:
  • Virtual proxy - Lazy loading (loading an image only when needed)
  • Remote proxy - Representing objects in another address space (RMI, gRPC)
  • Protection proxy - Access control (authorization, role checking)
  • Smart-reference proxy - Adds extra behavior (like logging, ref counting)
Flyweight
Flyweight pattern is a structural design pattern that reduces memory uses by sharing similar objects. Flyweight pattern uses a factory to manage these shared objects and ensures that each object is created only once. When we need large number of similar objects then creating and storing each object separately can take too much memory. Flyweight pattern offers solution by sharing similar objects to reduce memory uses and improve performance. Suppose we are building a text editor. Each character typed by a user is an object. If each character holds font, size, style and the character itself, then millions of characters would lead to high memory usage. Instead, we can store font, size, style (intrinsic state) in a shared Flyweight object and only keep position (x, y), selection, color (extrinsic state) in each instance.

Components:
  • Flyweight interface – Declares methods that accept extrinsic state.
  • ConcreteFlyweight – Implements Flyweight and stores intrinsic state.
  • FlyweightFactory – Manages a pool of flyweight objects.
  • Client – Maintains or computes extrinsic state and passes it to the flyweight.

Pros and Cons:
  • Reduces memory usage by sharing objects.
  • Increases performance when creating many similar objects.
  • Makes code more complex.
  • Requires careful separation of intrinsic and extrinsic state.

Real-world examples:
  • TextEditors (Word, Docs) - Characters reuse glyph data (shape, font) and only store unique positioning data.
  • Game - Thousands of similar objects like trees, rocks, or bullets share model/texture data using flyweights.
  • Map apps - Like Google Maps use flyweights to draw common icons (restaurants, pins, markers).
  • Doc Formats (PDF) - Rendering systems often use flyweights for fonts, styles, and reusable UI components.
Decorator
Decorator is a structural design pattern that allows us to add new behavior or functionality to objects dynamically without altering their structure or modifying their source code. This is achieved by creating decorator classes that wrap the original object and add new behaviours or attributes. Decorators provide a flexible alternative to subclassing for extending functionality. Decorator design pattern allows behaviours to be added to individual objects, either statically or dynamically without affecting the behaviour of other objects from the same class.

When-to use:
  • When we want to add responsibilities to individual objects, not an entire class.
  • When we want to keep classes open for extension but closed for modification.

Components:
  • Component: The interface or abstract class defining the operations.
  • Concrete component: The original object that we want to extend.
  • Decorator: Implements the same interface and has a reference to a Component object.
  • Concrete Decorators: Add new behaviors before or after delegating the request to the wrapped component.

Pros and Cons:
  • Adheres to SRP and OCP.
  • Greater flexibility than static inheritance.
  • Can add or remove responsibilities at runtime.
  • Many small objects, leading to complexity.
  • Harder to debug and maintain if overused.
  • Decorator logic may be spread out and hard to trace.

Real-world examples:
  • Java IO Streams.
  • Spring's Transactional, Cacheable can be seen as decorators.
  • Spring Framework - Middleware or Interceptors decorate request and response handling. 


                    InputStream istream = new FileInputStream("file.txt");  # Base Component
                    istream = new BufferedInputStream(istream);                # Decorator adds buffering
                    istream = new DataInputStream(istream);                     # Decorator adds data handling
Observer
Observer pattern is a behavioral design pattern that defines a one-to-many dependency between objects. In this pattern an object known as subject maintains a list of its dependent objects known as observers. When the subject's state changes, subject automatically notifies all its observers. The observer pattern defines a one-to-many dependency where, when one object's state changes, all objects that depend on it are notified and updated automatically.

When-to use:
  • Use Observer when you want to automatically notify multiple objects of state changes.
  • Great for event-driven systems, GUIs, distributed messaging, and real-time updates.
  • Think of it as a push-based notification system for objects.

Components:
  • Subject – The core object being observed. Maintains a list of observers.
  • Observer – Interface that defines the update method.
  • ConcreteSubject – Implements subject and notifies observers.
  • ConcreteObserver – Implements observer and responds to state changes.

Pros and Cons:
  • Loose coupling between subject and observers.
  • Easy to add or remove observers dynamically.
  • Promotes separation of concerns.
  • If many observers exist, notification may become slow.
  • Observers are not guaranteed to be in a specific order.
  • Difficult to debug due to indirect communication.

Real-world examples:
  • Stock market - Investor apps observe stock changes.
  • GUI Toolkits - Buttons and events use observer pattern internally.
  • YouTube-Instagram - Subscribers are notified when new content is posted.
  • News Feeds - Users follow a source, and get notified when updates happen.
Visitor
Visitor Pattern is a behavioral design pattern that lets us add new operations to existing object structures without modifying their classes. This pattern helps separate algorithms from the objects on which they operate. Visitor pattern is useful in scenario when an object structure is stable but we frequently need to extend its behaviours with new operations. This pattern aligns with open-closed concepts.

Components:
  • Visitor – Interface or abstract class for visiting elements.
  • ConcreteVisitor – Implements operations on concrete elements.
  • Element – Interface that defines accept(visitor)
  • ConcreteElement – Implements accept(visitor)
  • Client – Traverses object structure and applies visitor.

Real-world examples:
  • Filesystem utilities – Backup, search, indexing tools can visit folders and files.
  • Game engines – Collision detection or rendering can be implemented as visitors over game entities.
  • AST Traversal– Different visitors (semantic analyzer, code generator) operate on the same tree nodes.
  • Document converters – HTML, PDF, MD converters can visit document elements (paragraphs, images, links).

Pros and Cons:
  • Add new operations without changing existing object structures.
  • Encourages separation of concerns (data vs behavior).
  • Makes operations extensible and easy to manage.
  • Adding a new element (type) requires updating all visitors.
  • Involves double dispatch – one call from client to element, one from element to visitor.
Memento
Memento design pattern is a behavioral design pattern that allows us to capture and externalize an object's internal state so that it can be restored later, without violating encapsulation. Memento pattern helps store the state of an object and restores when needed. Commonly used to implement to undo-redo functionality. We can use memento design pattern when we need to capture and restore the state of an object without violating encapsulation. Command design pattern encapsulates actions, Memento encapsulates state.

Components:
  • Originator - The object whose state we want to save and restore.
  • Memento - A object that acts as a snapshot of Originator’s state. It is immutable and doesn’t expose internal details.
  • Caretaker - Object that knows when to save and restore Originator’s state but doesn't know the content of Memento.

Real-world examples:
  • TextEditors - Undo-redo functionality.
  • Games - Save and restore player progress.
  • Graphics Design - Photoshop history panel.
  • Databases - Rollback transactions.
  • Financial Apps - Restoring previous transaction data for audit.

Comparisons:
  • Command - Encapsulates actions, Memento encapsulates state.
  • State - Allows dynamic change of state, Memento stores past state.
  • Prototype - Clones the whole object while Memento saves only necessary internal state.

Drawbacks:
  • Makes rollback logic external and sometimes complex.
  • Requires proper management of mementos: Make Memento immutable.
  • Can consume a lot of memory if too many states are stored: Limit how many mementos are stored (undo levels).
Mediator
Mediator design pattern is a behavioral design pattern that promotes loose coupling between objects by centralizing communication between them. Instead of objects referring to each other directly, they communicate through a mediator object. This makes the system easier to maintain, modify, and scale, especially as the number of objects increases. Mediator pattern introduces a third-party object called the mediator to control the interation between two or more objects called colleagues. Mediator pattern focuses on simplifying communication between objects by managing and coordinating their interations through a mediator. When-to use: when you want to reduce complex dependencies between objects and promote decoupling. Mediator pattern is used for coordinating communication while the proxy pattern is used for controlling access.

Components:
  • Mediator interface: Declares the communication contract used by components.
  • ConcreteMediator: Implements coordination logic and knows all components (colleagues).
  • Colleague: Individual objects that communicate via the mediator. Knows about the mediator but not other colleagues.

Pros and Cons:
  • Promotes low coupling between components.
  • Simplifies maintenance and enhances reusability.
  • Centralizes complex communications.
  • Mediator can become a God object if it gets too complex.
  • Too much logic inside the mediator can lead to poor scalability.

Real-world examples:
  • Chat Room App: Chat room server = Mediator, users = Colleagues.
  • UI Components (MVC) - Dialog boxes, buttons send events to a central controller.

 
    interface ChatMediator {
        void addUser(User user);
        void sendMessage(String message, User user);
    }
    
    class ChatMediatorImpl implements ChatMediator {
        List users;

        void addUser(User user) {
            users.add(user);
        }
        
        void sendMessage(String message, User user) {
            for (User user: users)
                if (user != this)
                    user.receive("From {user.name}: {message}");
        }
    }

    abstract class User {
        ChatMediator mediator;
        String name;

        User(ChatMediator mediator, String name) {
            this.mediator = mediator;
            this.name = name;
        }

        abstract void send(String message);
        abstract void receive(String message);
    }


    class UserImpl extend User {
        UserImpl(ChatMediator mediator, String name) {
            super(mediator, name);
        }

        void send(String message) {
            println("{name} sends message: {message}");
            mediator.sendMessage(message, this);
        }

        void receive(String message) {
            println("{name} receives message: {message}");
        }
    }

    void main(String[] args) {
        ChatMediator mediator = new ChatMediatorImpl();
        User user1 = new UserImpl(mediator, "user1");
        User user2 = new UserImpl(mediator, "user2");
        User user3 = new UserImpl(mediator, "user3");
        User user4 = new UserImpl(mediator, "user4");
        User user5 = new UserImpl(mediator, "user5");
        mediator.addUser(user1);
        mediator.addUser(user2);
        mediator.addUser(user3);
        mediator.addUser(user4);
        mediator.addUser(user5);
        user1.send("Hello everyone.");
    }
Iterator
Iterator pattern is a behavioral design pattern that provides a way to access the elements of a collection (like a list or tree) sequentially without exposing its underlying representation. Iterator pattern is used to traverse elements in a container. Iterator pattern decouples container traversal algorithms from containers. Intent is to provide a standard way to traverse through a collection of objects without exposing its internal structure.

Components:
  • Iterator (Interface) – declares the interface for accessing elements.
  • ConcreteIterator – implements the Iterator interface and keeps track of the current position.
  • Aggregate (Interface) – declares a method to return an iterator.
  • ConcreteAggregate – implements the method to return an iterator instance.

Advantages:
  • Separates collection traversal logic from collection data structure.
  • Uniform traversal for various types of collections.
  • Supports multiple iterations simultaneously.


    interface Iterator {
        boolean hasNext();
        Book next();
    }

    interface Collection {
        Iterator iterator();
    }

    class BookStore implements Collection {
        List<Book> books;

        void addBook(Book book) {
            books.add(book);
        }

        List getBooks() {
            return books;
        }

        Iterator iterator() {
            return new BookStoreIterator();
        }
    }


        
    
    class BookStoreIterator implements Iterator {
        BookStore bookStore;
        int current;
        
        BookStoreIterator(BookStore bookStore) {
            this.bookStore = bookStore;
            this.current = 0;
        }
        boolean hasNext() {
            return current < bookStore.getBooks().size();
        }
        Book next() {
            return hasNext() ? 
            bookStore.getBooks().get(current++) : null;
        }
    }

    void main(String[] args) {
        BookStore bookStore = new BookStore();
        bookStore.addBook("Book1");
        bookStore.addBook("Book2");
        bookStore.addBook("Book3");
        Iterator iterator = bookStore.iterator();
        while (iterator.hasNext())
            println(iterator.next());
    }
Chain-of-responsibility
Chain-of-Responsibility design pattern is a behavioral design pattern that decouples sender and receiver of a request by connecting multiple handlers in a chain. The request is then passed along the chain until it is processed by one of the handlers.When-to use: when we have multiple objects that can handle a request but the specific object that handles it is determined at runtime.

Components:
  • Handler interface: Declares a method for handling requests and a reference to the next handler.
  • ConcreteHandler: Handles requests it is responsible for or forwards them to the next handler.
  • Client: Sends the request to the first handler in the chain.

Workflow:
  • A client sends a request to the first handler in the chain.
  • If the handler can handle it, it does.
  • Otherwise, it forwards the request to the next handler in the chain.
  • This continues until the request is handled or the end of the chain is reached.

Pros and Cons:
  • Reduces coupling between sender and receiver.
  • Adds flexibility in assigning responsibilities.
  • Chain can be modified dynamically.
  • Request may go unhandled.
  • Debugging can be tricky due to dynamic flow.
Command
Command design pattern is a behavioral design pattern that turns a request into a stand-alone object containing all the information about the request. This object can then be passed, stored, queued, and executed at a later time. Command pattern: an object is used to encapsulate all the information needed to perform an action or trigger an event at a later time, this information includes method name, the object that owns the method and the parameter for the method. Command pattern is perfect for features like scheduling, logging, undo and redo operations. When we need to encapsulate operations as objects and control behaviour through the arrangement of these operations. Command pattern encapsulates requests as objects, allowing the system to support complex command, request and operation management. Command pattern offers flexible way to extend system functionality while maintianing decoupling between the invoker and the receiver.

Components:
  • Command (Interface) – declares an interface for executing an operation.
  • ConcreteCommand – defines a binding between a receiver and an action.
  • Receiver – knows how to perform the operation associated with the command.
  • Invoker – asks the command to carry out the request.
  • Client – creates the command object and sets its receiver.

Pros and Cons:
  • Decouples objects invoking operations from those performing them.
  • Easy to add new commands without changing existing code.
  • Supports undo-redo, logging, and macro recording.
  • May result in lots of small classes ex. one per command.
  • Complexity may be overkill for simple operations.

Concepts:
  • Intent: Encapsulate a request as an object.
  • Useful-for: Decoupling sender and receiver, undo-redo, logging.
  • Real-world: Remote control, game commands, smart home actions.

Strategy
Strategy pattern allows us to define a set of algorithms and encapsulate them into independent classes so that they can be used interchangeably. Strategy pattern is a behavioral design pattern that separates the definition of algorithms from their usage, allowing the algorithms to vary independently of the client code. Strategy pattern is all about enabling flexibility in behavior by defining a family of algorithms, encapsulating each one, and making them interchangeable. Instead of hardcoding a behavior into a class, you delegate it to an object that implements a common interface. Core idea of this design pattern is to define a family of interchangeable algorithms.

When to use this pattern:
  • When have multiple algorithms (or behaviors) for a specific task.
  • When want to switch algorithms dynamically at runtime.
  • When want to avoid cluttering your code with long conditionals (like if-else or switch-case statements).

Components:
  • Strategy Interface: Common interface for all supported behaviors.
  • Concrete Strategies: Different implementations of the strategy.
  • Context: Uses a Strategy object and delegates it to the strategy interface.

Advantages:
  • Open-Closed Principle: Add new strategies without modifying existing code.
  • Cleaner Code: Eliminates repetitive conditionals.
  • Interchangeability: Strategies can be swapped out easily.


    interface PaymentStrategy {
        void pay(double amount);
    }

    class CreditCardStrategy implements PaymentStrategy {
        String cardNumber;

        CreditCardStrategy(String cardNumber) {
            this.cardNumber = cardNumber;
        }
        void pay(double amount) {
            println("Paid {amount} using Credit Card.");
        }
    }

    class PaypalStrategy implements PaymentStrategy {
        String email;

        PaypalStrategy(String email) {
            this.email = email;
        }
        void pay(double amount) {
            println("Paid {amount} using Paypal.");
        }
    }

    class UPIStrategy implements PaymentStrategy {
        String upiId;
        
        UPIStrategy(String upiId) {
            this.upiId = upiId;
        }
        void pay(double amount) {
            println("Paid {amount} using UPI.");
        }
    }


    class ShoppingCart {
        List items;

        void addItem(Item item) {
            items.add(item);
        }

        void removeItem(Item item) {
            items.remove(item);
        }

        double calculateTotal() {
            return items.stream()
            .mapToDouble(Item::getPrice).sum();
        }

        void pay(PaymentStrategy paymentStrategy) {
            double amount = calculateTotal();
            paymentStrategy.pay(amount);
        }
    }

    void main(String[] args) {
        ShoppingCart cart = new ShoppingCart();
        Item item1 = new Item("Item1", 200);
        Item item2 = new Item("Item2", 400);
        Item item3 = new Item("Item3", 700);

        cart.addItem(item1);
        cart.addItem(item2);
        cart.addItem(item3);
        
        cart.pay(new CreditCardStrategy("43GH-23BF-HGJF-B432"));
        cart.pay(new PaypalStrategy("koomar.dipu@gmail.com"));
        cart.pay(new UPIStrategy("6200035453@ibl"));
