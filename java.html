Java
Java is a high-level, general purpose, object-oriented, generic, concurrent and functional programming language. Java enables programmer to write once, run anywhere (WORA), meaning compiled java code can run on all platform without the need to recompile. Java applications are typically compiled to bytecode that can run on any JVM regardless of the underlying computer architecture. Java runtime provides dynamic capabilities such as reflection and runtime code modification that are typically not available in traditional compiled languages.

Object-oriented: Java is a complete object-oriented programming language.
Multithreaded: Java has support for multithreading at the language level.
Dynamic: Java support dynamic code modification through reflection.
Simple: Java is simple and easy to learn.
Portable: Java program is compiled to bytecode insted of machine code.
Secure: Because execution of bytecode is under the control of the JVM.
Compiled and interpreted: Java program is compiled then interpreted by the JVM.
High Performance:Java bytecode executes with high performance because of JIT.
Platform Independent: Java is independent of operating system platform.
Architecture Neutral: Java is independent of hardware platform.

Java Virtual Machine
Java virtual machine (JVM) is an abstract computing machine which loads, verifies and executes bytecode. Bytecode is the instruction set of the java virtual machine. Most well known JVM implementations are Java Hotspot Performance Engine and Eclipse OpenJ9.

Java virtual machine is divided into three main subsystem:
  • Class loader subsystem
  • Runtime data area
  • Execution engine

Class loader subsystem is responsible for loading, linking, verifying and initializing java class file. Runtime data areas contain class area, heap area, stack areas and PC registers. Execution engine contains an bytecode interpreter, JIT compiler and memory manager.

The class loader is part of the java virtual machine which is used to load class files during execution of a program. Application classes and all the dependencies from imported JARs are loaded by the system class loader.

  • Bootstrap class loader loads JDK core classes (rt.jar).
  • Extension class loader loads classes defined in lib\ext directory.
  • System class loader loads classes from the classpath.
  • Custom class loader loads classes from third party libraries.

Verify: Bytecode verifier will verify whether the generated bytecode is proper or not if verification fails we will get the verification error. Prepare: all static variables memory will be allocated and assigned with default values. Resolve: all symbolic memory references are replaced with the original references from method area.
Java Memory
Class Area: Class elements like constant pool, class fields, constructor codes, method codes are stored here. All executing threads share this part of the JVM memory area.

Heap: Heap is the area of memory where states of the each objects are stored. Heap memory is not thread-safe as it is shared by all threads of a program. Heap is dynamic area of memory.

Stack: Stack memory stores information about the active methods of a java program which grows and shrinks as new methods are added and removed to stack memory. Stack memory gets automatically allocated and deallocated. Access to stack memory is fast as compared to heap memory. Stack memory is thread-safe as each thread has its own stack memory.

Native Stack: Stack memory stores information about the active native methods.
PC Registers: PC registers stores the address of currently executing instruction.

Variable
Local variables:
  • Variable declared inside method is called local variable.
  • Local variables are stored on stack.
  • Local variables must be initialized before first use.
  • Local variables can only be accessed by the code inside the method.
  • Memory for local variable is allocated when corresponding method is called.
  • Local vars exist only while the method in which they are declared is executing.

Intance variables:
  • Instance variables are declared inside class and outside method.
  • Instance variables are initialized to their default values.
  • Instance variables are stored in the heap area of memory.
  • Instance variables represents state of the object.
  • Instance variables are accessed using object of the class.
  • Instance variables can not be accessed directly from static context.
  • Instance variables are created when an object of the class is created.
  • Instance variables are destroyed when the object is destroyed.
  • Each objects of a class have separate copy of the instance variables.

Static variables:
  • Static variables are created inside class and outside method with static keyword.
  • Static variables are stored in the class area of the java virtual machine.
  • Static variables are initialized only once when the class is loaded.
  • Static variables belongs to the class and not to object.
  • Static variables can be accessed directly by the class name.
  • Lifetime of an static variable is the same as the lifetime of the program.
  • Static variables can be accessed by both instance method and static method.
  • Memory allocation only happens once when the class is loaded in the memory.
Static Block and Instance Initializer
The static block is a block of statement inside a java class that will be executed when a class is first loaded into the JVM. A static block helps to initialize the static data members just like constructors helps us to initialize instance members. A static block executes once in the life cycle of any program, another property of static block is that it executes before the main method.

Instance initialization blocks are used to initialize the instance data members. Instance initialization blocks runs each time a new instance is created. Instance initialization blocks comes in the order in which they appear.

Order of execution:
  • Instance initialization block of the superclass.
  • Constructors of the superclass.
  • Instance initialization blocks of the class.
  • Constructors of the class.
Method
Methods are used to perform a simple task of the program. Method header provides interface to the other methods. Methods form the object's interface with the outside world. Objects define their interaction with the outside world through the methods that they expose.

Instance method is a method that belongs to instances of a class. Instance method defines behaviour of the objects. Instance method can be overriden, abstract and concrete. Instance method can refer to this and super keyword. Instance method can not be accessed from the static context. Instance method can access instance variables, static variables, instance methods and static methods.

Static method is a method which belongs to the class and not to the object. Static method can not access instance variables or methods of the class. Static method can only access static variables or methods of the class. Static method is associated with the class in which it is defined. Static method cannot refer to this or super keyword. Static method is statically bound to method call. Static method can be accessed directly by the class name without any object.
Wrapper Class
Wrapper classes provide a way to use primitive data types as objects. For each primitive types there is corresponding wrapper class. Wrapper classes resides in java.lang package. All of the wrapper classes are immutable. Generics can only accepts reference types. Java does not support template of primitive types. Wrapper classes are used extensively with Collection API classes.
Autoboxing and Unboxing
Automatic conversion of primitive types to corresponding wrapper class is known as autoboxing. Automatic conversion of wrapper class to corresponding primitive types is known as unboxing. Autoboxing and unboxing is performed at compile time by compiler.

Java compiler applies autoboxing when a primitive value:
  • Assigned to a variable of the corresponding wrapper class.
  • Passed as param to a method that expects an object of the corresponding wrapper class.

Java compiler applies unboxing when a wrapper class object:
  • Assigned to a variable of the corresponding primitive type.
  • Passed as param to a method that expects an argument of corresponding primitive type.
Access Modifiers
Access modifiers are used for defining accessibility of classes, methods and fields.

default: Declarations are visible only within the package.
private: Declarations are visible within the class only.
protected: Declarations are visible within the package or all subclasses.
public: Declarations are visible everywhere.

  • default: class is only accessible by classes in the same package.
  • public: class is accessible by any other class.
  • final: class cannot be inherited by other classes.
  • abstract: class cannot be used to create objects.

Method modifiers:
  • default: method is only accessible in the same package.

Field modifiers:
  • default: field is only accessible in the same package.
  • private: field is only accessible within the declared class.
  • protected: field is accessible in the same package and subclasses.
  • public: field is accessible for all classes.

  • static: field belongs to the class rather than an object.

Constructor
Constructor is a kind of method that is used to initialize newly created objects. Constructor has the same name as the class name in which it is declared. Constructor is automatically called upon creation of a new object. Constructor cannot be abstract, static, final and synchronized. Constructor can be overloaded. Constructor can not have return type. Constructor can be parameterized or can have zero parameter. Constructor with zero parameter is known as default constructor. Constructor overloading is the special kind of method overloading. Parent class constructor is called before the subclass constructor is called.
Constructor Chaining
Process of calling a constructor from another constructor. Expose multiple ways of constructing an object using constructors chaining. Constructor chaining makes our code more readable and avoid duplicate code.

Constructor chaining can be done in two ways:
  • By using this keyword for chaining constructors in the same class.
  • By using super keyword for chaining constructors from the parent class.
This and Super
This keyword is an implicit reference parameter of each instance method. This keyword refers to the current object in a method or constructor. This keyword contains address of the object that called the method. This keyword is an implicit parameter of each instance method. This keyword can be used to call current class constructor. This keyword can be used to call current class method. This keyword refers to the object that invoked the method. This keyword can be used to return the current class object. This keyword can be used to pass an argument in the method call. This keyword can be used to pass an argument in the constructor call.

The super keyword is a reference variable which refers to parent class object. The super keyword can be used to refer to parent class instance variable. The super keyword can be used to refer to parent class constructor. The super keyword can be used to refer to parent class method.

Exception
Exception is an unexpected event which occurred during the execution of a program and disrupts the normal flow of instructions. Exception object contains information about an error that happens at run time. Creating an exception object and handing it to the runtime system is called throwing an exception.

Exception can be divided into two board categories: faults introduced by the programmer, such as division by zero or calling a method with null reference and things out of the programmer's control, such as a user entering a garbage on the keyboard when the program expects a positive integer.
Checked Exception
Checked exceptions are those checked by the java compiler at compile time. Checked exceptions are forced to be caught or declared by the compiler. Checked exceptions are also called compile time exceptions. Checked exceptions can be handled. Checked exceptions can be propagated through throws. Code will not compile if checked exceptions are not handled. Checked exception represent a potential problem that the user knows can happen. Checked exception forces the user to establish an error handling policy at compile time.

Unchecked exceptions are not checked by the compiler at compile time. Unchecked exceptions are also called RuntimeExceptions or Error. Unchecked exceptions does not need to be caught or declared. Unchecked exceptions are caught by the JVM. Unchecked exceptions can not be handled. Runtime exception indicates an issue with the code that can be prevented. Unchecked exceptions represent an unexpected defect in the application that happens at runtime. Unchecked exceptions does not force the user to establish an error handling policy on compile.
Try-catch and Finally
Try, catch and finally blocks helps in writing the application code which may throw exceptions at runtime and gives us a chance to either recover from exception by executing alternate application logic or handle the exception to report back to
the user. It helps in preventing the ugly application crashes.

  • Code to be monitored for exceptions is written inside try block.
  • Catch block contains exception handling code.
  • We can use multiple catch with a single try.
  • Catch block must be written next to the try block.
  • Finally statement defines a code block to run regardless of the result.

Serialization
Serialization is the process of converting an object into stream of bytes. Once we have a stream of bytes we can store that into database, memory or into a file. Serialization is done to persist data or state of an object beyond the life cycle of an application. Second reason to serialize an object is to transmit data or state of an object over a network.

All subtypes of a serializable class are also serializable. Serializable interface has no methods or fields and serves only to identify the semantic of being serializable. Serializability of a class is enabled by the class implementing the io.Serializable interface. Classes that do not implement this interface will not have any of their state serialized or deserialized.

Multithreading
Multithreading is a process of executing multiple threads simultaneously. Multithreaded programs are capable of doing more than one tasks in parallel. Multithreading allows different parts of the program to execute simultaneously. Applications that perform several tasks that are not dependent on one another benefit the most from multithreading. Parallelism is about doing multiple tasks at once. Concurrency is about dealing with multiple tasks at once.
Thread
A thread is a single path of execution in a process. A thread has separate execution path. A path of execution is sequential execution of a set of instructions from start to end. Java application can have multiple threads running concurrently which can communicate with each other to achieve a common goal. Each java program has a special thread created by the JVM called the main thread. Thread class defines a thread but the code to be executed by the thread can be provided by any class that implements the Runnable interface.

Ways of creating threads:
  • Implement the Runnable interface and pass an instance of the class implementing the Runnable interface to the Thread constructor.
  • Extend Thread itself and override its run method.
Thread Priority
Every thread has a priority. The thread scheduler uses priority to determine which thread should be allowed to execute. Threads with higher priority are executed in preference to threads with lower priority.

When multiple threads are ready to execute then JVM will selects and executes the Runnable thread which has the highest priority. In case two threads have the same priority then JVM will execute them in FIFO order.

Thread State
New: A newly created thread that has not yet started the execution.

Runnable: Either running or ready for execution but waiting for resource allocation.

Blocked: Thread is waiting to acquire a monitor lock to enter (execute) a synchronized-block or synchronized-method.

Waiting: Thread is waiting for some other thread to perform a particular action without any time limit.

Timed waiting: Thread is waiting for some other thread to perform a specific action for a specified period.

Terminated: Thread has completed its execution.
Waiting and Blocking State
A thread goes to waiting state once it calls wait on an object or Thread.join method. A thread in the waiting state is waiting for a signal from another thread which will then remain in this state until another thread calls notify or notifyAll method. Once this thread is notified, it will not be runnable as it might be that other threads are also notified using notifyAll, so it will be in blocked state till it gets its chance.

A blocking state will occur whenever a thread tries to acquire lock on object and some other thread is already holding the lock. In the blocking state, a thread is about to enter a synchronized section but there is another thread currently running inside a synchronized block on the same object. The first thread must then wait for the second thread to exit block.
Interrupt
If any thread is in sleeping or waiting state (sleep or wait is invoked), calling the interrupt method on the thread, breaks out the sleeping or waiting state throwing InterruptedException. If the thread is not in the sleeping or waiting state, calling the interrupt method performs normal behavior and does not interrupt the thread but sets the interrupt flag to true.
Shared Data
When multiple threads try to modify a shared resource at the same time then the final outcome is non deterministic. We must explicitly synchronize concurrent access to a shared resource in a multi-threaded environment. When a thread enters synchronized code (normally by calling the synchronized method of that object) it takes the lock on that object and thus other threads cannot enters synchronized code because there is only one lock per object.

Suppose you are buying theatre tickets online and after you pick your seats and are entering your credit card information, someone else buys that seat and they are no longer available. Suppose multiple threads have access to a collection object and they are adding and removing at same time then there will be an error unless the collection is thread-safe.
Race Condition
Race condition occurs when two or more threads try to update mutable shared data at the same time. Two threads simultaneously reading data from the same object is not a problem but if both threads are simultaneously try to update the same object or if one thread is updating the object while the other is reading it then memory inconsistencies may arise.

  • Read-read sharing is safe
  • Both read-write and write-write races are a problem.
Deadlock
A deadlock occurs when there is more than one thread, each waiting for a resource held by another which occurs when multiple threads acquires the same set of lock in different order. The most obvious kind of resource is an object monitor but any resource that causes blocking such as wait-notify can qualify.

Deadlock happens when two or more threads become interlocked:
  • Thread A holds the lock in object1 but is waiting on object2
  • Thread B holds the lock in object2 but is waiting on object1
Thread Synchronization
Thread synchronization is a mechanism which ensures that two or more concurrent threads do not execute some particular section of program especially critical section. There are two types of thread synchronization mutual exclusion and inter-thread communication.

Following are some classic problems of synchronization:
  • Producer-consumer problem
  • Readers-writers problem
  • Dining philosopher problem
Mutual Exclusion
Mutual exclusion helps keep threads from interfering with each other while sharing data. Mutual exclusion ensures that only one thread can have access to shared data at a time. Mutual exclusion ensures that write functionality is thread-safe. Mutex is a program object which prevents simultaneous access to a shared data by multiple threads. Critical section is region of the code where shared data between multiple threads is accessed. Shared data is considered as a critical area.

Mutual exclusion can be achieved through:
  • Synchronized block
  • Synchronized method
  • Static synchronization
Monitor
Synchronization is built around an internal entity known as the monitor, monitors are used to achieved mutual exclusion and threads cooperation. A monitor is an entity that possesses a lock, a wait queue and an entry queue. Every Java objects has a monitor associated with them.

Lock enforces mutual exclusive access to synchronized code block or method invoked on the associated object. Wait set is the set of all the threads waiting on an monitor object that is used to provide coordination between multiple threads. Entry set is the set of all the threads who has already requested for the lock and the lock is not acquired by them yet.

A lock is a tool for controlling access to a shared resource by multiple threads. Every objects in java has an intrinsic lock associated with it. Only one thread at a time can acquire the lock and all access to the shared resource requires that the lock be acquired first. When a new thread tries to acquire the lock and if already a thread owns the lock then that thread will be waiting on the entry set to acquire the lock. When the thread which is acquired the lock completes execution of its critical section then it will release the lock. Next thread will acquire the lock but this next thread is taken from the entry set and will be determined by JVM based on some criteria like FIFO.
Volatile Keyword
Volatile keyword instructs JVM to read the value of the volatile variable from main memory and write value of the volatile variable to main memory. Volatile keyword instructs JVM to do not cache value of the variable locally so that changes to that variable must be seen by all subsequent reads by other threads regardless of the synchronization. Volatile keyword provides visibility just like synchronization but scoped only to each read or write of the field. Volatile keyword ensures that read functionality is thread safe. Visibility means changes done by one thread are visible to other threads which ensures that read functionality is thread safe.
Synchronized Keyword
A block of code marked with synchronized becomes a synchronized block. Keyword synchronized enforces exclusive access to the block of code. Only one thread is allowed to enter this block of code at the same time. Thread that first enters the synchronized block remains in runnable state. Synchronized blocks and synchronized methods are mutually exclusive. Synchronized method can be static method or instance method. The synchronized keyword does not support fairness. Fairness property provides a lock to the longest waiting thread (contention). Any thread can acquire lock once released, no preference can be specified. All other threads accessing the same block of code enters the blocked state. When a runnable thread exits the synchronized block then lock is released. Another thread is now allowed to enter the runnable state and place a new lock.
Synchronized Method
When two or more threads modify an object then declare the methods that carry out
the modification as synchronized. When a class has one or more synchronized methods, each object of the class gets a queue that holds all threads waiting to execute one of the synchronized methods. When a thread wants to execute a synchronized method on a given object, first it has to acquire the lock of that object. Once thread acquired the lock then it is allowed to execute any synchronized method on that object. When a thread calls a synchronized method on an object then the JVM ckecks the monitor lock for that object. If the monitor lock is unowned then ownership is assigned to the calling thread which is then allowed to proceed with the method call. If the monitor lock is owned by another thread then the calling thread will be put on hold until the monitor lock becomes available. When a thread exits a synchronized method it releases the monitor lock, allowing a waiting thread (if any) to proceed with its synchronized method call.
Cooperation
Cooperation is achieved using the wait set which is associated with the monitor and wait-notify mechanism. Cooperation is important when one thread needs some data to be in a particular state and another thread is responsible for getting the data into that state.

When a thread invokes the wait method respect to an object then the thread will be suspended and added to the wait set to wait until some other thread invokes notify or notifyAll on the same object. If a thread was put in the queue by a call to wait then it must be unfrozen by a call to notify before it can be scheduled for execution again. Notify method is used for waking up threads that are in the wait set of the monitor of a particular object. A thread may call wait, notify or notifyAll on an object only if it owns the monitor lock of that object.
Wait, Notify and NotifyAll
Wait: Causes the current thread to wait until another thread invokes the notify() method or the notifyAll() method for this object. The thread releases ownership of this monitor and waits until another thread notifies threads waiting on this object's monitor to wake up either through a call to the notify() method or the notifyAll() method.

Notify: Used to notify one thread that condition on which it is waiting has been satisfied. For all threads waiting in the wait set the method notify notifies anyone of them to wake up randomly. Wakes up a single thread that is waiting on this object's monitor. If many threads are waiting on this object then one of them is chosen to be awakened. A thread waits on an object's monitor by calling one of the wait methods.

NotifyAll: Used to notify one or more threads that conditions they are waiting on has been satisfied. Wakes up all threads that are waiting on the wait set of this object's monitor. A thread waits on an object's monitor by calling one of the wait methods. The awakened threads will compete in the usual manner with any other threads that might be actively competing to synchronize on this object. This method should only be called by a thread that is the owner of this object's monitor.
Locks
Synchronized keyword forces all lock acquisitions and releases to occur in a block structured way, meaning when multiple locks are acquired they must be released in the opposite order and all locks must be released in the same method in which they were acquired. Lock is acquired and released implicitly.

The reentrant lock provides more flexibility which allows a lock to be acquired and released in different scopes and allows multiple locks to be acquired and released in any order. Lock is acquired and released using lock and unlock methods.

Reentrant locks provides a convenient tryLock method which acquires lock only if it is available and not held by any other thread. This reduces the blocking of thread waiting for lock. The tryLock method backs out if the lock is not available immediately or before a timeout expires. Disadvantages: programmer is responsible for acquiring and releasing lock and introduces boilerplate code.
Thread Pool
The server programs such as database and web servers can simultaneously serves multiple clients. An approach for building a server application would be to create a new thread each time a request arrives and service this new request in the newly created thread. While this approach seems simple to implement, it has significant disadvantages. A server that creates a new thread for every request would spend more time and consume more system resources in creating and destroying threads than processing actual requests.

Thread pool represents a group of worker threads that are waiting for the job and reused many times. A thread pool manages a set of unamed threads that perform work on request. Threads do not terminate right away. When one of the threads completes a task, the thread becomes idle, ready to be dispatched to another task. The submitted tasks are processed in order, usually faster than could be done by creating a thread per task. The overhead of creating and destroying threads is limited to creating and destroying just the number of active worker threads in the pool.
ExecutorService
The Executor and more expansive ExecutorService interfaces define the contract for a component that can execute tasks. Users of these interfaces can get a wide variety of implementation behaviors behind a common interface.

The most generic Executor interface accepts jobs only in the form of Runnables:
  • void execute(Runnable command)

The ExecutorService extends Executor to add methods that take both Runnable
and Callable task and collections of tasks. 

  
		Future<?> submit(Runnable task)
		Future<T> submit(Callable<T> task)
		Future<T> submit(Runnable task, T result)
		T invokeAny(Collection<? extends Callable<T>> tasks)
		List<Future<T>> invokeAll(Collection<? extends Callable<T>> tasks)
Object Relational Mapping
Object-relational mapping (ORM) is the process of mapping java objects to relational tables of a relational database. Each row in the table correspond to the single object and each column in the table correspond to the each field of the object.

ORM is an layer between object-oriented language and SQL language of relational database. ORM layer work like a adapter layer as it adapts the language of object graphs to the language of relational tables. Data is structured into object graphs in the object oriented program and relational tables in the relational database.
