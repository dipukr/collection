System Design
System design is the process of defining the architecture, components, interfaces, and data of a system to meet specific requirements. It's a blueprint that guides the development and implementation of a system, ensuring it's functional, scalable, reliable, and maintainable. System design is a crucial process that ensures a system is well-structured, efficient, and can meet the evolving needs of its users. Key concepts include load balancing, caching, database sharding and understanding the CAP theorem.
Distributed Computing
Distributed computing involves multiple computers working together over a network to solve a problem, acting as a single, powerful system. It's a divide-and-conquer approach where tasks are broken down and processed concurrently on different nodes (computers), then the results are combined. This allows for greater scalability, performance, and resilience compared to using a single, centralized computer. The components of a distributed system communicate and coordinate their actions by passing messages to one another in order to achieve a common goal. Three significant challenges of distributed systems are: maintaining concurrency of components, overcoming the lack of a global clock, and managing the independent failure of components. When a component of one system fails, the entire system does not fail. Examples of distributed systems vary from SOA-based systems to microservices to massively multiplayer online games to peer-to-peer applications. Distributed systems cost significantly more than monolithic architectures, primarily due to increased needs for additional hardware, servers, gateways, firewalls, new subnets, proxies, and so on. There are many different types of implementations for the message passing mechanism, including pure HTTP, RPC connectors and message queues.

Key Concepts in Distributed System Design:
  • Nodes or components: These are the individual computers or processes that make up the distributed system.
  • Network: A network connects these nodes, enabling communication and data exchange.
  • Communication: Many protocols and mechanisms (REST, RPC, MQ) are used for communication between nodes.
  • Transparency: Users should not be aware of the underlying complexity of the distributed nature of the system.
  • Scalability: The system should be able to handle increasing load and data volume by adding more nodes.
  • Availability: The system should remain operational and accessible even if some nodes fail.
  • Consistency: Ensuring data consistency across multiple nodes is a significant challenge.
  • Fault Tolerance: The system should be designed to handle failures gracefully and continue functioning.
  • Security: How secure should the system be? Consider factors like authentication, encryption, and access controls.
  • Durability: How durable should the system be? What is the timeframe for data retention.
Scalability
Scalability is the ability of a system to handle an increased workload without loosing performance. The ability of a system to handle a growing amount of work by adding resources. The ability of a system to handle increasing traffic, data, or users without breaking or slowing down.
CAP
CAP Theorem asserts that in a distributed system, you can only guarantee two out of three: consistency, availability and partition tolerance.
Latency and Throughput
Latency measured in milliseconds (ms) represents the time taken for a single query to execute. Throughput measured in operations per second or transactions per second represents the volume of data processed per unit time. Latency focuses on the speed of individual operations while throughput focuses on the volume of operations that can be handled per unit time. Latency is the time it takes to process one request; throughput is how many requests are handled per second.
Cache
Caching is a techniques of storing frequently accessed data in a temporary storage location to improve the system performance. Caching helps to reduce the time and resources required to fetch the data from the original source by serving it quicky from the cache. Caching is a technique that allows frequently accessed data to be stored in memory so that it can be quickly retrieved without having to go back to the database every time. Cache reduces latency and load on the database. Web browser caches web pages to load them faster on subsequent visits. Database server caches frequently accessed query results to avoid expensive database queries.

Cacheable data: data that does not changes at all are the best candidate for caching and data that does not frequently changes can also be cached for certain period to time. Cache performs well when there is more cache-hits than cache-miss. Embedded cache Client-server cache: cache is between the application and the database. Also called shared cache or standalone cache. Consistent Hashing
A hashing technique that minimizes the number of keys that need to be remapped when a hash table is resized, useful in distributed systems. Consistent hashing is a distributed hashing algorithm designed to address the problem of data redistribution when nodes are added or removed. With traditional hashing algorithms adding or removing node often result in alot of data needing to be distributed which can lead to system instability and performance degradation. Goal of consistent hashing is to minimize the need for rehashing when the number of nodes in a system changes. Consistent hashing is used in distributed systems, such as web caches, CDNs and DBs to efficiently distribute data or load across a variable number of servers or nodes.
CDN
A geographically distributed network of proxy servers that work together to provide fast delivery of Internet content. A network of servers that delivers static content (like images or videos) from locations closer to the user.
DNS
Domain name system is a hierarchical and distributed name service that provides a naming system for computers, services, and other resources on the Internet or other Internet Protocol (IP) networks.

GeoDNS (Geographical Domain Name System) is a DNS service that redirects users to different servers based on their geographical location, optimizing traffic flow for better performance and availability. It works by identifying a user's IP address, determining their geographic location using IP geolocation databases, and then returning the IP address of a server that is closest or most appropriate for that region.
API Gateway
API gateway acts as a single entry point for handling client requests to various backend services. It acts as a reverse proxy, routing requests to the appropriate services and can also handle cross-cutting concerns like authentication, authorization, rate limiting and caching, streamlining API management and enhancing security. API gateway accepts API requests from a client, processes them based on defined policies, directs them to the appropriate services, and combines the responses for a simplified user experience. Typically, it handles a request by invoking multiple microservices and aggregating the results. Load balancer
Load balancing is the method of distributing network traffic equally across a pool of resources that support an application. Modern applications must process millions of users simultaneously and return the correct text, videos, images, and other data to each user in a fast and reliable manner. To handle such high volumes of traffic, most applications have many resource servers with duplicate data between them. A load balancer is a device that sits between the user and the server group and acts as an invisible facilitator, ensuring that all resource servers are used equally. Companies usually have their application running on multiple servers. Such a server arrangement is called a server farm. User requests to the application first go to the load balancer. The load balancer then routes each request to a single server in the server farm best suited to handle the request. Benefits of load balancer: load balancer improves an application’s availability, scalability, security and performance. We can use load balancing to make these tasks easier: Run application server maintenance or upgrades without application downtime.

SAGA
The saga pattern is a design pattern used in microservices architecture to manage distributed transactions. It allows for a sequence of local transactions across multiple services, ensuring data consistency even when individual services might fail. It achieves this by defining a series of compensating transactions that can be executed to undo previous operations if a failure occurs. There are two main ways to implement a saga: choreography and orchestration. Choreography: services react to events published by other services, deciding whether to perform their local transaction or a compensating transaction based on the event. Orchestration: a central coordinator service manages the saga, instructing each service when to perform its local or compensating transaction. Compensating transactions: for each local transaction, a corresponding compensating transaction is defined. If a transaction fails, the saga executes these compensating transactions in reverse order to undo the changes made by the successful transactions. Events-messages: Saga pattern relies on asynchronous communication (e.g., message queues) to trigger subsequent transactions and compensating actions, ensuring loose coupling between services.

Session and Cookies
Cookie data is stored on the client’s computer. Cookies can be set to expire after a specific time period or when the browser is closed. In some cases, it is essential that all requests from a client are sent to the same server for the duration of a session, for example when a client is putting items in a shopping cart and then completing the purchase. Maintaining the connection between client and server is known as session persistence. Without session persistence, data has to be synchronized across servers and potentially fetched multiple times, creating performance inefficiencies. Session data is stored on the server.

ACID
These four properties define the reliability and integrity of database transactions. They ensure that transactions are processed reliably and that the database remains in a consistent state, even in the face of system failures. Transactions are treated as a single, indivisible unit of work. Either all operations within a transaction are completed successfully, or none of them are. If any part of the transaction fails, the entire transaction is rolled back to its previous state, preventing partial updates. A transaction must bring the database from one valid state to another. This means that any transaction must respect all defined constraints, rules, and restrictions of the database. Concurrent transactions should not interfere with each other. Each transaction should be isolated from others, as if they were running sequentially. This prevents issues like dirty reads (reading uncommitted data) or lost updates. Once a transaction is committed, its changes are permanent and survive system failures. The database must guarantee that the data will be written to persistent storage and will not be lost due to power outages or other crashes.
Connection pooling
Connection pooling means that connections are reused rather than created each time a connection is requested. A connection pool is cache of database connections maintained so that connections can be reused when future requests to the database is required.

Sequence of steps involved in a typical database connection life cycle:
  • Opening a connection to the database using the database driver
  • Opening a TCP socket for reading-writing data
  • The user is authenticated
  • Reading-writing data over the socket
  • Close the connection
  • Close the socket
Sharding
Sharding is the process of dividing a database table (dataset) into smaller, more manageable pieces called shards, which are distributed across multiple machines or nodes. Sharding usually refers to splitting one large table’s data horizontally (row-wise) across multiple databases or servers. Each shard contains a subset of rows of the table. All shards have the same schema, just different data. Splits large datasets into smaller chunks (shards) to spread the load across servers.
Software Architecture and Design
Architecture of a system describes its major components, its relationships and how they interact with each other. Software architecture is the high-level, strategic blueprint of a system, defining its overall structure, core components, and their interactions to meet business needs, while design is the tactical, low-level process of detailing how individual components will be built and work together. Think of architecture as defining the what and why (using a microservices pattern for scalability), whereas design is the how of specific implementation, like choosing a particular design pattern (Strategy pattern) to solve a problem within one of those services.

  • Key decisions:
    • Choosing an overall pattern like microservices or serverless.
    • Addressing non-functional requirements like security, performance, and scalability across the system.
    • Defining the interactions between major components and systems.
Client-server model
The client-server model is a computing model where client devices request resources or services from a centralized server, which then processes the request and provides a response. Client-server model also called request-response model in which client sends a request and has to wait for the server's response before sending another one. In this model there is no continuous flow (live streaming) of data. For update client has to ask the server, server itself does not provides realtime updates.
