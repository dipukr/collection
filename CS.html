Monitor
Synchronization is built around an internal entity known as the monitor. Monitors are used to achieved mutual exclusion and threads cooperation. A monitor is an entity that possesses a lock, a wait-queue and an entry-queue. Every Java objects has a monitor associated with them. Lock enforces mutual exclusive access to synchronized code block or method invoked on the associated object. Wait set is the set of all the threads waiting on an monitor object that is used to provide coordination between multiple threads. Entry set is the set of all the threads who has already requested for the lock and the lock is not acquired by them yet.
Lock
A lock is a tool for controlling access to a shared resource by multiple threads. Every objects in java has an intrinsic lock associated with it. Only one thread at a time can acquire the lock and all access to the shared resource requires that the lock be acquired first. When a new thread tries to acquire the lock and if already a thread owns the lock then that thread will be waiting on the entry set to acquire the lock. When the thread which is acquired the lock completes execution of its critical section then it will release the lock. Next thread will acquire the lock but this next thread is taken from the entry set and will be determined by JVM based on some criteria like FIFO.
Visibility
Visibility ensures that read functionality is thread safe. Volatile ensures that all threads sees the same value of a variable at any given time.
Mutual exclusion
Mutual exclusion or atomicity ensures that write functionality is thread safe. Mutual exclusion is a property of concurrency control, which is instituted for the purpose of preventing race conditions. Race condition occures when a shared resource is accessed by different actors at the same time. Both mutex and semaphore helps managing access to shared resources but they do it in different ways. Mutex is suited for exclusive access whereas semaphore is suited for limited concurrency.
Thread
A thread of execution is the smallest sequence of programmed instructions that can be managed independently by a OS scheduler. Threads of a process share its executable code and the values of its dynamically allocated variables and non-thread-local global variables at any given time. Local primitive variables are thread-safe because each thread has separate stack memory. Local reference variables are not thread-safe because each thread share heap memory. A thread is a sequence of instructions within a program that can be executed independently of other code. Threads share memory (code and data) space.
Thread Synchronization
Thread synchronization is a mechanism which ensures that two or more concurrent processes or threads do not simultaneously execute some particular program segment known as critical section. Processes' access to critical section is controlled by using synchronization techniques. When one thread starts executing the critical section (serialized segment of the program) the other thread should wait until the first thread finishes. There is no any guarantee about exactly when a blocked thread is unblocked when the synchronized block is free again. This is typically up to the OS or execution platform to coordinate the unblocking of blocked threads.

Heap
Heap is a region of free memory that is allocated and deallocated at run time by the program.
ORM and JPA
Object-relational mapping (ORM) is the process of mapping java objects to relational tables of a relational database. Each row in the table correspond to the single object and each column in the table correspond to the each field of the object.
ORM is an layer between object-oriented language and SQL language of relational database. ORM layer work like a adapter layer as it adapts the language of object graphs to the language of relational tables. Data is structured into object graphs in the object oriented program and relational tables in the relational database.
Java Persistence API is the java standard API for object-relational mapping. JPA is one of the many possible approach to ORM. The API is a set of concepts that defines which objects within the application should be persisted and how it should persist them. JPA is only an specification and that it needs an implementation to work. Most popular implementation of JPA is Hibernate. JPA is concerned with persistence, which means any mechanism by which java objects outlive the application process that created them.
